Coding Agent 祖师爷——Claude Code
和业界许多 Coding Agent 前辈一样，DeerCode 受到 Claude Code 逆向工程的启发。考虑到国内外的编程模型可能针对 Claude Code 做过许多“特殊训练”，因此本文的工具设计也参考了它的设计。Coding Agent 的本质是一个 ReAct 风格的 Agent，它拥有自己的规划、感知和执行能力，通过调用只读的文件系统工具集获得仓库的结构和必要的上下文（Repo Structure Understanding），从而定位到与用户需求相关的文件和行号（Code Locating），然后再通过文本编辑器工具集将代码写入仓库（Code Generating），全程通过 To-do 列表工具进行分解、规划、监督任务进度。一些现代化的 Coding Agent 还支持 MCP 扩展、命令扩展（即 / 开头的命令集）和自定义 Instructions（比如我们熟悉的 .cursorrules、AGENT.md）等进行进一步的能力扩展。

ReAct 是 Reasoning and Act 的缩写，是一种让 LLM 能够“边思考边行动”的提示范式，核心思想是让模型交替进行 Reasoning（推理）和 Acting（行动）。
基本流程：
1. Thought（思考）：模型分析当前状态，决定下一步做什么
2. Action（行动）：执行具体操作（比如调用搜索 API、读文件）
3. Observation（观察）：获取行动结果
4. 重复 1-3，直到得出最终答案
举个例子：
User: 北京今天天气怎么样？

Thought: 我需要查询北京的实时天气信息
Action: search("北京天气 今天")
Observation: 北京今天晴，温度 15-25°C

Thought: 已经获取到天气信息，可以回答了
Answer: 北京今天晴天，温度在 15-25°C
为什么有用？
- 传统 LLM 是"一次性输出"，ReAct 让模型能多步骤解决复杂问题，比如能够及时通过调用工具获取（retain）外部环境的变化，并紧接着通过工具操纵（mutate）外部环境，并依次循环往复直到问题被解决或因异常而终止。
- 可解释性强：每一步推理过程都可见
- 适合需要外部工具调用的场景（搜索、计算器、数据库查询等）2.3 Single 还是 Multi-Agent？
关于 Coding Agent 是一个 Single Agent 还是 Multi-Agent，业界一直没有统一的答案。

笔者认为，目前来看相对比较理想的形态是 Claude Code Agent 的范式（如上图所示），它从外观层来看是一个 Single Agent，是一个标准的 ReAct loop 循环设计模式。当遇到复杂问题时，主 Agent 会调用 Task 工具，该工具会“裂变”出一个能力和自己几乎一模一样的 Sub-agent，并被分配到相同的工具链，并被指派一个更明确的子任务。Sub-agent 可以被用于未知的 Code RAG 探索过程、复杂逻辑的编写、独立 UI 组件的开发、文档编写等，在其执行完的结果会被并入到主链路中。Sub-agent 拥有自己独立任务目标、规划和执行，这相当于是分担了主 Agent 的任务和上下文，从而降低了对单 Agent 的能力上限要求，提升了 Claude Code 整体的能力天花板。

为何 Coding Agent 如此青睐 Console UI 设计？
你可能不禁要问，为何几乎所有的 Coding Agent 都清一色的选择了命令行 CUI（Console UI） 、CLI（Command Line Interface）呢？
- 轻量与低开发成本：对比图形化界面（GUI），Console UI 不需要处理复杂的窗口管理、按钮渲染、事件监听等，简单布局一下，保持一定的“程序员审美”的同时，开发效率高很多。
- 跨平台兼容性：终端几乎是所有开发环境的“公共语言”。无论是 Linux、macOS，还是 Windows，都能跑命令行。
- 适合持续开发集成工作流：可以被轻松的集成到 CI/CD 流程。
- 可移植性与低资源占用：Console UI 占用内存和 CPU 极小，非常适合在远程服务器、容器或嵌入式设备上运行，还可以通过 SSH 命令远程控制。
- 渐进式增强：可以先用 Console UI 实现 MVP，然后再推出 IDE 插件、Web 界面、桌面 App 等。
