## 第一部分 · 概念摸底（开场必问，决定第一印象）

### Q1. 什么是 agent？它和"直接调用 LLM"或"固定工作流(workflow)"的区别是什么？ ⭐⭐⭐

**【考点】** 你对 agent 的本质理解，是不是只会调 API 套壳。

**【回答结构】**
- 一句话：**agent = LLM + 一个让它能"观察环境、调用工具、根据结果决定下一步"的循环**。直接调 LLM 是"一问一答、无行动能力"；workflow 是"路径由人写死的 DAG"；agent 是"**下一步做什么由模型自己决定**"。
- 三者光谱：单次调用 → 链式/workflow（人定流程）→ agent（模型定流程）。区别的核心是**控制权在谁手里**：workflow 的分支是工程师写的 if-else，agent 的分支是模型每一步的自主决策。
- 代价对应：自主性越高，越不可控、越要 harness 兜底（校验、审批、压缩、取消、软着陆）。

**【高光】** 点出"workflow 适合流程确定的场景，agent 适合流程开放、需要探索的场景；不是越 agent 越好"——体现你懂取舍，而不是无脑追 agent。

---

### Q2. 解释一下 ReAct。为什么 agent 要用"循环"？ ⭐⭐⭐

**【考点】** 是否真懂 agent 的运行机制。

**【回答结构】**
- ReAct = **Reasoning + Acting** 交替：模型先推理(think)决定调哪个工具，执行(act)拿到观测(observation)，把观测喂回去再推理，直到它认为不需要再调工具、直接给最终答案。
- 为什么必须循环：模型一次推理拿不到现实世界的反馈。**"列目录→看到文件→决定读哪个→读到内容→回答"**，每一步都依赖上一步的真实结果，无法一次规划到底。循环就是"看一步、做一步、再看一步"。
- 落到我项目：`Agent.stream` 就是 `for step in 1..maxSteps: think → 若有 tool_use 则 act → 否则结束`。

**【高光】** 强调"模型的 tool_use 输出只是**意图**，真正的执行和观测发生在 harness 里，再回灌"——很多人讲 ReAct 会漏掉"观测回灌"这一环。

---

### Q3. 什么是 agent 的 harness？模型之外，工程上你能控制的有哪些？ ⭐⭐⭐

**【考点】** 这是 agent 岗位的核心词。能不能讲清"模型之外的那一圈工程"。

**【回答结构】**
- 一个 agent 产品 = **模型 + harness**。模型是黑盒，你只能选型、调采样参数、开关 thinking；**harness 是模型之外你能完全掌控的一切工程**。
- 列举我的 harness 覆盖面：怎么组织 prompt（稳定前缀）、怎么把工具暴露给模型、怎么执行工具并裁剪结果、怎么把观测喂回去、上下文涨了怎么压缩、危险操作怎么审批、出错怎么分级处理、长任务怎么软着陆、怎么取消。
- 一句话价值主张：**同一个模型，harness 的好坏决定它是玩具还是生产力工具。**

**【高光】** 这句"同模型、harness 决定上限"是这个岗位最想听到的世界观，务必说出来。

---

### Q4. LLM 是无状态的，那 agent 的"记忆"是从哪来的？ ⭐⭐⭐

**【考点】** 对上下文机制的底层理解，是后面所有 context engineering 问题的地基。

**【回答结构】**
- LLM 每次调用都是独立的、无状态的——它不"记得"上一轮。
- 所谓记忆**全是工程拼出来的**：每一步 think 都要把 **system 提示 + 完整对话历史 transcript + 工具 schema** 整段重新发过去。模型的"连续性"是 harness 每轮重新塞进去的。
- 由此推出三个根本性质（后面所有问题都从这推）：上下文是**有限**的（会撑爆）、**昂贵**的（不变前缀被重复计费几十次）、**承载注意力**的（噪音多会带偏模型）。

**【高光】** "记忆是工程拼出来的幻觉"——把它说成第一性原理，然后顺势引出"所以才需要缓存（省钱）、压缩（防爆）、子 agent（控噪音）"，把后面的题串成一条线。

---

### Q5. 什么是 context engineering？和 prompt engineering 什么关系？ ⭐⭐

**【考点】** 区分两个常被混用的概念。

**【回答结构】**
- prompt engineering 是**写好一段提示词**（措辞、few-shot、格式）；context engineering 是在"每步重发整段对话"这个约束下，**经营整个上下文窗口这块有限、昂贵、承载注意力的预算**——决定什么进窗口、占多少、何时清理。
- 它是 multi-turn、长任务 agent 才有的问题：单次问答没有"上下文增长"。
- 我项目里的五个手段：稳定前缀、prompt caching、逐工具的 tool-result 裁剪、compaction、子 agent 隔离 + skills 渐进披露。**贯穿主线一句话：把上下文当预算来经营，按需支付。**

**【高光】** 用"预算"这个比喻统领全部手段，面试官会觉得你有体系而非零散记忆。

## 第二部分 · 项目深挖（简历写了 trendpower，必然逐条问）

### Q6. 用两三分钟介绍一下你这个项目 / 整体架构。 ⭐⭐⭐

**【考点】** 电梯陈述能力 + 架构感。这是项目深挖的总开关，答好了引导面试官往你准备充分的方向追问。

**【回答结构】**（控制在 90 秒）
- 定位：从零实现的**生产级 ReAct agent 框架（harness）**，纯 Python 核心库无界面依赖，配终端 UI 和浏览器可视化，约 1 万行。
- 分层：三个包**严格单向依赖** `web → tui → core`；核心库内部 `coding → agent → foundation` 四层，第三方集成走横向 `community`。**核心完全不知道界面是否存在。**
- 能力面：异步流式 ReAct 循环 + 可插拔中间件 + 工具系统（16 个工具）+ 完整上下文工程（缓存/压缩/子agent/渐进披露）+ 协作式取消 + 多厂商 Protocol 适配。
- **主动抛钩子**：结尾留一句"其中我最想讲的是缓存和压缩之间一个隐藏耦合的 bug"——把面试官往你最强的 Q10 引。

**【高光】** 最后那个主动埋点，让你掌握深挖节奏。

### Q7. 你的 ReAct 循环为什么用 async generator（异步生成器）？里面那个"哨兵值"是干嘛的？ ⭐⭐

**【考点】** 对并发/流式控制流的掌控，Python 异步功底。

**【回答结构】**
- 为什么流式：agent 的价值有一大半在"过程可见"。循环本身是 async generator，一边推进 think→act，一边 `yield` 出 `progress`（思考中/正在跑某工具）和 `message`（助手回复、工具结果）事件，UI 能 `async for` 实时渲染，而不是转圈等几十秒。
- 代价：调用方必须 `async for` 消费、必须处理中途取消——但对交互式 agent 值。
- 哨兵值：异步生成器有个语言限制——它能流式 `yield`，但**不能像普通函数那样既 yield 又 return 一个最终值**（Python 没有 TS 的 `yield* + return value`）。我用一个内部哨兵事件 `{"type": "_think_done", ...}` 把"这一步 think 的最终结果"从生成器流里带出来，外层循环识别这个哨兵、取出结果继续 act。

**【高光】** 补一句并发是"白送"的：`_act` 用 `asyncio.create_task` + `asyncio.wait(FIRST_COMPLETED)`，同一轮多个工具**真正并行**，并发是架构自带的不是额外加的。

### Q8. 中间件架构怎么设计的？为什么 compaction 中间件要 insert(0) 放最前面？ ⭐⭐

**【考点】** 解耦能力、对"顺序即语义"的理解。

**【回答结构】**
- 问题：压缩、技能注入、待办、审批这些都是**横切关注点**，写进主循环会让循环腐化成巨型函数。
- 设计：主循环只管 think/act，在固定生命周期节点（`beforeModel/afterModel/beforeToolUse/afterToolUse/beforeAgentRun...`）广播事件；中间件是**鸭子类型**对象，用 `getattr(mw, hook, None)` 探测——实现哪个钩子就挂哪个，只关心自己那部分。
- 三条约束：
  1. **顺序即语义**：中间件是有序列表。compaction `insert(0)` 是因为它必须在其它读 transcript 的 `beforeModel` 钩子（如 skills 注入）**之前**先把历史压实——否则别的中间件读到的是没压缩的超长历史。
  2. **返回值即修改**：钩子返回 dict 被 merge 进共享的 `AgentContext`，这是影响主流程的**唯一**途径，边界清晰。
  3. **可组合**：四个自带中间件互不感知、自由增删。

**【高光】** 直接对标 Web 框架中间件 / 洋葱模型，说"这是经过验证的扩展点形态"。

### Q9. 工具的输入来自模型，是不可信的；你怎么处理？ ⭐⭐⭐

**【考点】** "模型输出不可信"这个第一性原则，agent 健壮性的核心。

**【回答结构】** 讲"校验—执行—规整—裁剪"四段式：
1. **校验闸门**：`define_tool` 用 **pydantic** 模型在执行前强制校验模型吐出的原始 dict。缺字段、类型错、幻觉参数——根本进不了执行体；校验失败自动变成一条工具错误喂回模型让它**自我修正**，而不是抛异常崩掉。
2. **执行**：执行体在"输入已合法"前提下干活，逻辑简化。
3. **规整**：结果统一成结构化形态——成功 `{ok, summary, data}`、失败 `{ok:false, summary, error, code}`，并把 code 归类成 `invalid_input/not_found/environment_missing`，让模型**按错误类型决策下一步**。
4. **裁剪**：按工具分级裁剪后才进对话（见 Q18）。
- 逃生舱：`raw_input_schema` 让 MCP 外部工具直接透传自己的 JSON Schema，不经 pydantic 往返。

**【高光】** 一句话拔高：把"不可信输入 + 任意副作用 + 不可控输出"系统性驯化成"可信、稳定、可控的一条消息"——这是 harness 健壮性的核心。

### Q10. ⭐⭐⭐【重点准备】缓存和压缩之间那个隐藏耦合的 bug，详细讲讲。

**【考点】** 这是你简历的王牌，最能证明"真正跑通过 agent、对系统全局数据流有掌控"。面试官一旦追问到这，讲好了基本稳了。**务必背熟"约束→设计→冲突→修复"四段叙事。**

**【回答结构】**
1. **背景约束**：模型无状态，每步要重发 system + 完整对话 + 工具清单。开头那一大段不变的前缀被一字不差重发几十次，**又贵又慢**。
2. **两个对策本身**：
   - **缓存（省钱省延迟）**：把不变内容堆在请求最前形成稳定前缀，在三个稳定边界打 Anthropic 缓存断点（工具数组尾、system 块、最后一条消息的最后一块）。不变前缀走缓存、增长的对话走增量缓存。
   - **压缩（防撑爆）**：对话超阈值时摘要中段，靠 `promptTokens` 判断"对话是否过长"。
3. **冲突点（高光）**：开缓存后，Anthropic 返回的 `input_tokens` **只统计未命中缓存的部分**，命中的单列在 `cache_read_input_tokens` / `cache_creation_input_tokens`。于是 `promptTokens` 会因为缓存命中而**虚假变小** → 压缩以为对话还很短 → **压缩永不触发 → 最终撑爆上下文窗口**。两个看似无关的模块，通过一个共享数值 `promptTokens` **远程耦合**了。
4. **修复与验证**：在 token 统计处把缓存部分**加回去**，恢复"本轮 prompt 真实大小"的语义，并在"一次性返回 `_to_token_usage`"和"流式累计 `StreamAccumulator`"**两条路径保持一致**：
   ```python
   prompt_tokens = input_tokens + cache_read + cache_creation
   ```

**【高光】**
- 这是"两个模块通过共享量远程耦合"的真实案例，能定位它证明你对全局数据流有掌控。
- 补一刀对比：OpenAI 系（含豆包）是自动前缀缓存、`prompt_tokens` 本就含缓存部分，所以这个 bug 在那条路径**不存在**——说明你理解不同厂商计费语义的差异，而不是死记一个 fix。

### Q11. 上下文压缩(compaction)具体怎么做？里面最容易踩的坑是什么？ ⭐⭐⭐

**【考点】** 长任务的核心能力，"问答对不可分割"这条不变量只有真跑通 agent 才会踩到。

**【回答结构】**
- 触发：`estimate_tokens` 超过 `trigger_tokens`（默认约 100k）时触发，是个 `beforeModel` 钩子。
- 策略：**保头 + 保尾 + 摘中段**。保头（任务目标/项目信息，丢了模型就忘了在干嘛）、保最近 N 轮（当前进展，逐字保留）、把中间一大段探索细节用模型摘成一小段原地替换。摘要提示明确要求保留：目标、改过的文件及状态、命令及结果、决策及理由、未解问题、下一步。
- **最大的坑——tool-pairing 不变量**：带 `tool_use` 的 assistant 消息后面**必须**紧跟对应 `tool_result`，provider 会拒绝落单的任何一半。所以 `plan_compaction` 切分时绝不让 head 收尾在一个其结果会被摘掉的 `tool_use` 上，也绝不让 tail 起始于孤儿 `tool` 消息——用两个 while 循环把切口往安全处回退。
- 触发估算的稳健性：优先用真实 `promptTokens`，回退到字符启发式 `len/4`，**取较大值**——这样刚产生一波大工具结果也能立即触发，不必等下一次模型响应。（顺势可回扣 Q10 的耦合。）
- **失败安全**：摘要调用抛错就记 warning、`return None` 不压缩继续跑——**绝不让辅助功能搞挂主任务**。

**【高光】** "问答对不可分割"是只有真正跑过 agent 才会踩到的正确性不变量；"取较大值"和"失败兜底"体现边界条件的扎实把握。

### Q12. 你做了子 agent，它解决什么问题？和压缩什么关系？有哪些安全约束？ ⭐⭐⭐

**【考点】** 上下文隔离、多 agent 的取舍、安全意识。

**【回答结构】**
- 动机：像"在整个 repo 里查清某功能怎么实现"会翻几十个文件、产生几十条 grep/read 噪音。若留在主线，烧 token、稀释主 agent 注意力、还破坏缓存前缀。
- 设计：`task` 工具把脏活派给一个**独立子 agent**（自己的 transcript + 受限工具集），干完**只回传一句结论**，中间过程主线完全看不到。两种模式：`explore`（只读工具、免审批）和 `general`（完整工具、转发父级审批）。
- **和压缩的关系（高光）**：子 agent 是**事前隔离**（噪音根本不进主线），压缩是**事后有损补救**（噪音已经在主线里了再摘要）。两者互补：一开始就知道是大段探索 → 用子 agent；噪音已混进主线（如一长串交互式调试）→ 靠压缩兜底。
- **两条写死的安全红线**：子 agent **永不含 `task`**（递归深度恒为 1，杜绝无限套娃，靠在追加 `task` 进主工具集**之前**截取 base_tools 快照实现）、**永不含 `ask_user_question`**（后台任务不能反向弹窗阻塞用户）。
- 并发白送：主循环并行 act，一次发 3 个 `task` 就是 3 个子 agent 并发跑。

**【高光】** "用一次额外模型开销，换主线上下文的纯净度"——把它定性为一种上下文工程手段，而不只是功能。

### Q13. 交互式 agent 要能随时打断，你的取消机制怎么做的？为什么不能直接 kill？ ⭐⭐

**【考点】** 并发控制、资源清理、"协作式 vs 抢占式"的理解。

**【回答结构】**
- 机制：自定义 `AbortSignal`，把"打断"从用户一路串到底层。循环每步开头 `throw_if_aborted` 主动检查；`_act` 里把工具任务和一个 `abort_task` 用 `asyncio.wait(FIRST_COMPLETED)` **竞速**——取消信号一到立刻抛 `AbortError`、未完成的工具 task 被 cancel；`bash` 工具注册 `proc.kill` 回调在取消时杀子进程；子 agent 注册 `inner.abort` 联动停止。
- 为什么协作式而非强杀：**没有谁被强杀**，所有人监听**同一个信号**各自优雅退出、各自清理资源（杀子进程、关连接）。强杀会留下僵尸进程、半写文件。
- 为什么要竞速而不是 `await gather`：若只 `await gather(tasks)`，取消要等所有工具自然结束才生效——一个慢命令就让 Ctrl-C 失灵。

**【高光】** "竞速 abort_task 让取消**立即**生效"这一点最能体现你懂交互式系统的体验底线。

### Q14. 长任务跑满步数上限(maxSteps)了怎么办？直接报错吗？ ⭐⭐

**【考点】** 失败处理的成熟度，用户体验意识。

**【回答结构】**
- 多数实现的做法：直接 `raise`，丢光全部进展——这是最差体验（跑了几分钟、改了一半文件，只甩一句报错）。
- 我的做法——**软着陆**：撞 `maxSteps`（默认 100）时注入一条提示要求模型"停止调工具，总结已完成/遗留/下一步建议"，并把这一轮的 `tools` 临时置为 `None` **强制只出文字**，返回这份交接总结。
- 代价仅一次额外模型调用；换来"我做到这一步、剩下这些、建议接着这么做"，用户可据此续问或开新一轮。
- 这是更大的"失败分级"哲学的一部分：工具出错→变成 `Error:` 消息喂回模型**自愈**；压缩失败→降级不压缩继续；撞上限→软着陆；前端再兜一层（`AbortError` 静默、其它异常变友好提示）。

**【高光】** "强制 tools=None"保证收尾轮一定产出文字而不会又跑去调工具——这个细节体现你真踩过坑。

### Q15. 怎么支持多个模型厂商？接一个新的国产模型成本多大？ ⭐⭐

**【考点】** 抽象能力、"面向接口而非实现"。

**【回答结构】**
- 核心只依赖一个 `ModelProvider`——Python 的 **`Protocol`（结构化类型）**，只要实现 `invoke`/`stream` 就是合法 provider，**无需继承**。
- 因此：接 Anthropic、OpenAI 各写一个适配类；而豆包/火山方舟、DeepSeek、Qwen、Kimi、GLM 等**兼容 OpenAI 接口的厂商连新类都不用写**——直接复用 `OpenAIModelProvider`，配置里加个预设（base_url + 模型名）即可。
- 统一的 `Message` 类型是"对话真相的唯一来源"，provider 只在边界做格式翻译（如把 tool_result 翻成 Anthropic 要求的 user 角色块）。换厂商时循环、工具、中间件**一律不动**。

**【高光】** "一个恰当的 Protocol 换来对整个国产大模型生态的零成本兼容"——是"面向抽象编程"的活教材。

---

### Q16. 工具结果可能很大（一次 grep 刷屏），你怎么控制它占多少上下文？ ⭐⭐

**【考点】** tool-result 工程，对"每个 token 都是成本和注意力"的理解。

**【回答结构】**
- 原则：**不同工具的结果按信息密度分别对待**，把"摘要 vs 原始数据 vs 截断长度"做成按工具配置的一等公民。
- 具体策略：
  - `list_files/glob/grep` → `preferSummaryOnly`，只留一句摘要丢弃原始 data（模型要的是"找到了什么"，不是把整棵目录树背进上下文）。
  - `read_file` → 保留内容但上限约 12000 字符（读文件本就是为了看内容）。
  - 写类工具 → 中等上限。
  - `task` 子 agent 报告 → summary-only、上限约 8000（整个内层 run 的浓缩，值得多留）。
- 价值：在**源头**控制上下文的增速和信噪比。

**【高光】** 主动暴露一个已知可改进项：bash 超长输出目前"留头丢尾"，但测试/堆栈的关键信息常在结尾——这是值得继续打磨的点。**敢讲不足，反而显得你真的在思考而非背稿。**

### Q17. prompt caching 的断点为什么打在那三个位置？ ⭐

**【考点】** 对缓存机制的细节理解（Q10 的延伸追问）。

**【回答结构】**
- 缓存的前提是**稳定前缀**：请求字节布局刻意设计成"越稳定的越靠前"——`[工具 schema(整个 run 不变)] [system 提示(不变)] [增长的对话]`。
- 三个断点：**最后一个工具**（缓存整个工具 schema 数组）、**system 块**、**最后一条消息的最后一个块**（让不断增长的对话也走增量缓存，下一轮命中）。
- 配套关键决策：**AGENTS.md（项目指令）放进 system 提示而非首条 user 消息**——这样它既进入可缓存的不变前缀，又**永远不会被 compaction 误删**（压缩只改写对话消息、不动 system）。skills 也是把 `<skill_system>` 追加到 prompt 末尾、基串不变，保证每轮拼出的 system 逐字节相同、可缓存。

**【高光】** "把 AGENTS.md 放 system 同时拿到缓存 + 防压缩两个好处"——一个决策解决两个问题，体现设计的经济性。

## 第三部分 · 系统设计 / 开放题（考综合与迁移能力）

### Q18. 让你从零设计一个 coding agent，你会怎么设计？ ⭐⭐⭐

**【考点】** 把项目经验抽象成方法论的能力。这题就是让你把 trendpower 的设计"反过来讲成方法论"。

**【回答结构】**（自顶向下）
1. **核心循环**：ReAct，流式（过程可见、可取消）。
2. **工具层**：定义工具集（读写文件、执行命令、检索）；**入参用 schema 强校验**（不可信输入在边界驯服）；**结果结构化 + 分级裁剪**。
3. **扩展机制**：横切关注点（压缩/审批/技能）做成**有序中间件**，不焊进循环。
4. **上下文工程**：稳定前缀 + 缓存省钱、tool-result 策略控增速、compaction 兜底防爆、子 agent 隔离噪音、技能渐进披露。
5. **健壮性**：协作式取消、失败分级（工具错自愈 / 撞上限软着陆 / 辅助功能失败降级）。
6. **可替换后端**：Protocol 抽象 provider。
7. **（主动补 eval）**：上线前要有可观测性和 eval 闭环（见 Q20）。

**【高光】** 按"循环→工具→扩展→上下文→健壮→后端→评估"七层讲，体现体系化思维。每层都能落到 trendpower 的具体实现当论据。

---

### Q19. 一个长任务的上下文眼看要爆了，你有哪些手段？怎么排优先级？ ⭐⭐

**【考点】** context engineering 的综合应用题。

**【回答结构】** 按"事前预防 → 过程控制 → 事后补救"排：
- **事前**：稳定前缀 + 缓存（不省窗口但省钱省延迟）；**子 agent 隔离**让探索噪音根本不进主线；skills 渐进披露按需加载。
- **过程中**：**tool-result 分级裁剪**，从源头压低每条观测的体量（摘要 vs 原文 vs 截断）。
- **事后**：**compaction** 保头保尾摘中段兜底；触发判断取真实/粗算较大值，保证大工具结果能立即触发。
- 优先级：能事前隔离的优先隔离（子 agent）> 控制单条增速（tool-result）> 实在满了才有损压缩（compaction）。**有损压缩是最后手段，因为它会丢信息。**

**【高光】** "把上下文当预算经营、按需支付"这条主线再次出现，前后呼应。

### Q20. 怎么评估一个 agent 好不好？你项目里怎么做 eval 的？ ⭐⭐

**【考点】** 工程闭环意识。**注意：我的项目目前没做完整 eval（只有 46 个单测覆盖核心路径），这题要诚实 + 给出方案，不能硬编。**

**【回答结构】**
- 诚实定位：项目目前用 **46 个单元/集成测试**覆盖循环、压缩、子 agent、缓存等核心**正确性**路径；系统级 eval 是我 backlog 上的项，还没做完整闭环。
- 如果做，我会分两层：
  1. **离线 eval**：建任务集（每个任务有可验证的成功判据，如"测试通过""文件改对"），跑 agent 统计**任务成功率、平均步数、token 成本、延迟、缓存命中率**；用 LLM-as-judge 评主观质量。
  2. **可观测性支撑**：先做 **per-step tracing 落盘**（每步的 prompt 大小、工具调用、延迟、token、缓存命中率）——没有 trace 就没法定位"为什么这个任务失败/变贵"，eval 才能驱动提示和策略迭代。
- 关键指标取舍：成功率是结果，但要配上**成本/步数**——一个靠堆 token 提成功率的 agent 不是好 agent。

**【高光】** 诚实承认未做 + 给出有条理的方案 + 强调"没有 tracing 就没有 eval"，比硬吹一个不存在的 eval 系统强得多。面试官非常吃这套。

### Q21. 什么时候该用多 agent，什么时候单 agent 就够？ ⭐

**【考点】** 不盲目追多 agent 的判断力。

**【回答结构】**
- 默认**单 agent**：协调成本、上下文传递损耗、调试难度都低。
- 上多 agent（子 agent）的判据：**子任务有大量中间噪音、且只需回传一句结论**（如全仓检索、深度调研）——这时隔离的收益（保护主线上下文）大于多跑一个 agent 的开销。
- 我的 `task` 工具就是这个判据的落地：`explore` 模式专门给只读、高噪音的探索类子任务。
- 反例：流程紧耦合、需要频繁来回交互的子任务，拆成多 agent 反而因为上下文割裂而更差——这种该留在主线或用 workflow。

**【高光】** "多 agent 不是越多越好，核心判据是'噪音能否被隔离 + 结论能否被浓缩'"。

---

### Q22. 怎么降低 agent 的成本和延迟？ ⭐

**【考点】** 生产意识。

**【回答结构】**
- **缓存**：稳定前缀 + 缓存断点，把不变前缀的重复计费和 prefill 延迟压掉——harness 里 ROI 最高的单点改动。
- **控上下文增速**：tool-result 分级裁剪、子 agent 隔离，少喂 token 直接省钱。
- **模型分级**：贵模型做主推理、便宜模型做摘要/分类（我 backlog 里"用更便宜的模型做 compaction 摘要"就是这个思路）。
- **并发**：同一轮多工具并行，压墙钟时间。
- **失败少走弯路**：结构化错误让模型一次纠对，而不是反复试错烧 token。

**【高光】** 把"成本/延迟"拆成"省重复（缓存）、省体量（裁剪/隔离）、省单价（模型分级）、省时间（并发）、省弯路（结构化纠错）"五个维度，显得有方法论。

## 第四部分 · 反思与权衡（考深度与诚实）

### Q23. 这个项目你觉得最难 / 踩坑最深的地方是什么？ ⭐⭐⭐

**【考点】** 真实经历，最能区分"真做过"和"背简历"。

**【回答结构】** 首推两个：
- **缓存↔压缩耦合 bug**（Q10）：最难的不是写缓存或写压缩，而是发现"两个独立模块通过一个共享数值远程耦合"。它在短对话里完全不暴露，只有长 run 才撑爆——是排查"为什么长任务偶尔爆窗口"时一路顺着 token 统计逆推才定位的。
- **压缩的 tool-pairing 不变量**（Q11）：第一版压缩直接按消息数切，结果 provider 时不时拒绝整个请求——才意识到 `tool_use`/`tool_result` 必须成对，切口不能劈开问答对。

**【高光】** 这两个都是"只有真正跑通过 agent 才会踩到"的坑，讲它们 = 证明你不是玩具级 demo。

### Q24. 如果重做，哪里会不一样？还有什么没做？ ⭐⭐

**【考点】** 自我批判和对完整性的认知。诚实列 backlog。

**【回答结构】** 直接讲 backlog（不影响已有设计成立，是锦上添花）：
- **bash 工具加固**：执行超时防永久挂起、stdout+stderr 双路捕获、超长输出"留头+留尾"（现在留头丢尾，关键信息常在尾）、结构化 exit code。
- **Provider 重试退避**：对 429/5xx/网络抖动做指数退避，避免一次瞬时错误崩掉长 run。
- **Compaction 提速降本**：用更便宜的模型做摘要、或后台预压缩，避免阻塞当轮。
- **可观测性/tracing + eval 闭环**（见 Q20）。
- **联网工具**（web_search/web_fetch）让研究类 skill 名副其实。
- **环境上下文块**：system 前缀里加稳定 `<env>`（cwd 树/git 状态/平台/日期），帮模型定向且进缓存零额外成本。

**【高光】** "这些是清单上有优先级的待做项"——体现你对'完成'和'完善'有清醒边界，而非自我感觉良好。

### Q25. 你反复强调"单向依赖""核心不知道界面存在"，这带来的实际好处能举个证据吗？ ⭐

**【考点】** 架构原则不是口号，要有可验证的回报。

**【回答结构】**
- 证据一：`trendpower-web` 要加"把每轮 LLM 请求可视化"，**只需在最薄的 provider 层继承重写 `stream`/`invoke`、广播请求**，完全不碰循环、工具、中间件。
- 证据二：核心库可独立运行（`examples/`），不拉任何 UI 依赖；agent 层领域无关（不懂"编程"只懂"想—做"），能驱动任意领域 agent。
- 证据三：MCP 集成按同一原则拆分——通用协议客户端**下沉**核心 `community/mcp/`，进程相关接线（配置路径、生命周期、`/mcp` 命令）**上浮**前端，前端 import 核心。

**【高光】** "边界清晰的回报是可验证的扩展自由"——用三个具体改动证明抽象选对了，而不是空谈解耦。
