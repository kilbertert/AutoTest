# 第 11 课 · 设计哲学与扩展指南

最后一课，把前面十课的零件升华成"设计观"，再给你四张**"想加功能从哪下手"的速查表**。学完这一课，你不只是"看懂了 trendpower"，而是"能上手改造它"。

## 一、回看：三条贯穿全局的设计纪律

### 纪律 1：分层 + 单向依赖

trendpower 从大到小都在贯彻"**下层不依赖上层、依赖只能单向**"：

```
包级：   trendpower-web → trendpower-tui → trendpower(core)
核心内： coding → agent → foundation     （community 横向供给集成）
```

**为什么值得**：
- 核心库**不知道界面存在**，所以同一个内核能包成终端、网页、或别的（第 10 课已亲眼见到 web 只在 provider 层切一刀）。
- `agent` 层**不懂编程**，只懂"想-做"，所以这套循环能驱动任意领域的 agent，不只是 coding。
- 加集成（新模型、MCP）只在 `community` 层动，不污染上面。

一句话：**边界划清楚，每一块都能独立替换、独立测试、独立演进。**

### 纪律 2：把"上下文"当预算经营

这是 context engineering 那条主线（第 7 课），值得再钉一遍：

> 上下文窗口是**有限、昂贵、承载注意力**的预算。trendpower 的每个相关决策都在回答"这笔预算花得值不值"——稳定前缀省钱、缓存省钱、工具结果裁剪控增长、压缩兜底、子 agent 隔离、技能渐进披露按需付费。

### 纪律 3：辅助功能绝不连累主任务

第 9 课那条哲学：可恢复的错喂回模型自愈、辅助功能（压缩）失败就退化但不崩、实在到头了优雅交卷。**健壮性不是事后补的，是每个零件设计时就带的态度。**

## 三、四张扩展速查表（想加功能从哪下手）

### 速查表 A：加一个**工具**

> 例：加一个 `http_get` 工具。

1. 在 `coding/tools/` 新建 `http_get.py`。
2. 用 pydantic 定义参数（带 `Field(description=...)`，那是给模型的说明书）：
   ```python
   class _HttpGetParams(BaseModel):
       description: str = Field(description="为什么要发这个请求")
       url: str = Field(description="要请求的 URL")
   ```
3. 写异步 `_invoke`，**返回结构化结果**（`ok_tool_result(...)` / `error_tool_result(...)`，第6课）。
4. `define_tool(name="http_get", ...)` 导出。
5. 在 `coding/agents/lead_agent.py` 的 `tools = [...]` 列表里加上它。
6. 要审批就把名字加进 `permissions/requires_approval.py`；要给子 agent 用就注意 `task.py` 的只读/通用分组（第8课）。
7. 给结果加裁剪策略？改 `agent/tool_result/policy.py`（第6课）。

### 速查表 B：加一个**模型 provider**（接新厂商）

> 大多数国产模型兼容 OpenAI 接口，**根本不用写新 provider**——直接在 `model_providers.py` 加一个预设、类型填 `"openai"` 即可（豆包就是这么接的，第10课）。

只有当某家接口**既不兼容 OpenAI、也不兼容 Anthropic** 时，才需要：
1. 写一个类，实现 `foundation/models.py:ModelProvider` 这个 Protocol 的两个方法：`invoke` 和 `stream`（第3课）。
2. 在里面把 trendpower 的 `Message[]` 翻译成该家 API 的格式、再把回复翻译回 `AssistantMessage`（参考 `community/anthropic/utils.py`）。
3. 别忘了上下文工程那套：如果该家支持缓存，比照 `community/anthropic/model_provider.py` 打缓存标记 + 折算 token（第7课）。
4. 在前端 `app.py:_try_create_runner` 里按配置选用它。

### 速查表 C：加一个**中间件**（加横切行为）

> 例：每步把对话大小打到日志。

1. 写一个工厂函数，返回一个带相应钩子的 `SimpleNamespace`（第5课）：
   ```python
   def create_logging_middleware():
       async def before_model(params):
           print("当前对话条数：", len(params["agentContext"]["messages"]))
           return None      # 不改任何东西就返回 None
       return SimpleNamespace(beforeModel=before_model)
   ```
2. 插进 `lead_agent.py` 的 `middlewares` 列表。**注意顺序**——要在压缩之后读对话就排它后面，要最先跑就 `insert(0)`（第5课"顺序即语义"）。

### 速查表 D：加一个**技能**（灌入最佳实践）

> 完全不用写代码！

1. 在技能目录（仓库根目录的 `skills/`）建一个文件夹，比如 `my-skill/`。
2. 里面放一个 `SKILL.md`，开头写 frontmatter（名字 + 简介）：
   ```markdown
   ---
   name: my-skill
   description: 一句话说清这个技能干嘛、什么时候该用
   ---
   （正文：详细的工作流程、步骤、注意事项……）
   ```
3. 完事。技能中间件开场会自动发现它，把"名字+简介"注入目录；模型判断相关时自己 `read_file` 读正文（第8课的渐进披露）。

## 四、还在演进的方向（设计是活的）

再好的设计也不是终点。trendpower 优化清单上还有（教学诚实，也示范"怎么看待一套设计的不足"）：

- **bash 工具加固**：加执行超时（防永久挂起）、stdout+stderr 都抓、超长输出"留头+留尾"。
- **provider 重试退避**：一次网络抖动/限流不该崩掉整个长任务。
- **压缩提速降本**：用更便宜的模型做总结、或后台预压缩。
- **可观测性**：把每步的 token/延迟/缓存命中落盘，支撑离线评估。

这些都不影响前面讲的设计成立——它们是"锦上添花"，也提醒你：**好的架构，是让这些改进都能在不动主干的前提下加上去。**（试着用速查表想想每一项该动哪儿？)

## 五、结业小结

你现在应该能回答最开始那个问题了——**"换我来设计，为什么也会这么做"**：

- 因为 agent 要"看一步做一步"，所以有 **ReAct 循环**；
- 因为有些行为是横切的，所以有**可插拔中间件**；
- 因为模型输出不可信，所以工具有 **pydantic 校验边界**；
- 因为模型每步重发整桌纸，所以要**缓存、裁剪、压缩、隔离**；
- 因为会出错、会被打断，所以有**取消信号和失败三件套**；
- 因为想让内核能复用、能包成多种产品，所以有**分层和单向依赖**。

每一个"为什么"，都是一个真实约束逼出来的、合理的工程决策。这就是 harness engineering 和 context engineering 的全部。
