# 第 9 课 · 健壮性与收尾

前面每一课都埋了一句"这条暗线第 9 课收"。现在把散落各处的**"出错了怎么办、取消怎么办"**串成一条线。一个 agent 跑几十步、调一堆工具、连网等模型，出岔子是常态——**健壮性决定了它是"一出错就崩"还是"稳稳兜住"。**

---

## 一、取消：从用户一路传到子进程被 kill（端到端）

### 设计

交互式 agent 必须能随时打断，而且要**立刻**生效——不能等当前慢命令跑完。trendpower 用第 3 课认识的 `AbortSignal`（急停按钮）把"打断"从用户一路传到最底层。

### 端到端走一遍

```
用户按 Ctrl+C
  → AgentRunner.abort()              （前端，trendpower-tui/.../agent_runner.py）
  → agent.abort()                    （Agent 持有的"按钮"被按下）
  → AbortController.abort() → 信号触发
       ├─ 循环每步开头 signal.throw_if_aborted() → 抛 AbortError，循环中止
       ├─ _act 里的 abort_task 赢得赛跑 → 立刻 throw_if_aborted()
       └─ bash 工具注册的 _kill 回调被触发 → proc.kill() 杀掉子进程
```

逐段印证代码：

**① 按钮和信号**（第 3 课见过）：

```python
# foundation/abort_signal.py
class AbortSignal:
    def throw_if_aborted(self):      # 已被按下就抛 AbortError
        if self._aborted: raise ...AbortError...
    async def wait(self): await self._event.wait()    # 可被 await 等待（第1课A2的Event）
    def add_listener(self, cb):      # 注册"被按下时执行的回调"
        if self._aborted: cb(); ...  # 注意：如果已经按下了，立刻执行
        else: self._listeners.append(cb)
```

**② 循环里检查 + 赛跑**（第 4 课见过）：

```python
# agent/agent.py
self._abort_controller.signal.throw_if_aborted()     # 每步开头查一次
...
abort_task = asyncio.create_task(signal.wait())      # _act 里和工具任务赛跑
if abort_task in done: signal.throw_if_aborted()     # 急停先到 → 抛错中止整轮
```

**③ 工具底层响应**（第 6 课见过）：

```python
# coding/tools/bash.py
remove_listener = signal.add_listener(lambda: proc.kill())   # 急停 → 杀子进程
try:
    await proc.communicate()
finally:
    remove_listener()                                # 善后解绑（第1课A7）
```

**④ 子 agent 也连动**（第 8 课见过）：

```python
# agent/subagent/runner.py
signal.add_listener(inner.abort)     # 主线被急停 → 子 agent 的循环也被急停
```

**一条信号，串起了用户、主循环、工具、子进程、子 agent。** 这就是"协作式取消"——没人能强杀谁，但大家都监听同一个信号、各自优雅退出。

## 二、失败三件套：崩、兜底、还是优雅收尾？

不同的"出错"，trendpower 用不同的态度对待。这是设计成熟度的体现。

### 1）工具出错 → 不崩，变成消息喂回模型

单个工具抛异常（参数校验失败、文件不存在、命令报错），**不让它掀翻整个循环**：

```python
# agent/agent.py:_act 里的 run_one（简化）
try:
    result = await tool.invoke(tool_use["input"], signal)
except Exception as error:
    result = f"Error: {error}"      # ★把异常变成一条结果★
```

这条 `"Error: ..."` 会被规整成结构化错误（第 6 课的 `errorKind`）喂回模型，让它**看到错误、自我修正**——比直接崩有用得多。

打个比方：实习生干活搞砸一步，正确做法是告诉他"这步错了，错在这"让他重来，而不是当场把整个项目叫停。

### 2）撞步数上限 → 不崩，软着陆

防死循环设了 100 步上限。撞到时，**不报错**，而是让模型用现有信息交一份总结：

```python
# agent/agent.py:_emit_step_limit_summary（简化）
async def _emit_step_limit_summary(self):
    self._append_message({"role": "user", "content": [{"type": "text", "text": _STEP_LIMIT_PROMPT}]})
    # _STEP_LIMIT_PROMPT: "你已到步数上限，不能再调工具。请总结：做了什么、还差什么、下一步建议。"
    saved_tools = self._context.get("tools")
    self._context["tools"] = None          # ★强制只能出文字，不能再调工具★
    try:
        async for ev in self._think():     # 跑最后一轮（无工具）
            ...
    finally:
        self._context["tools"] = saved_tools   # 善后还原
    ...
```

设计巧思：把 `tools` 临时设成 `None`，模型这一轮**没工具可调**，只能输出文字总结——保证收尾轮一定产出答案而不会又去调工具。

打个比方：实习生干大活到下班还没完，让他交一份"做到这儿了、剩这些、建议这么接着干"的交接，而不是甩一句"超时了"摔门走人。

> 这是最近才从"硬崩溃"改成"软着陆"的——见项目优化记录。

### 3）压缩失败 → 不崩，继续不压缩

第 7 课见过：总结那一步要是出错了，记条日志、**返回 `None` 继续跑**，绝不让"压缩"这个辅助功能反而搞挂正常任务。

```python
# agent/compaction/compaction.py
except Exception as exc:
    _log.warning("压缩失败，继续不压缩: %s", exc)
    return None
```

### 三件套的共同哲学

> **辅助功能出错，绝不连累主任务；可恢复的错，变成信息喂回去让模型自愈；实在到头了，优雅交卷而非崩溃。**

### 前端也兜一层

桥接器 `AgentRunner`（第 2 课）在最外面再兜一层（`trendpower-tui/.../agent_runner.py`）：
- 捕获 `AbortError` → **静默**（用户主动取消，不算错误，不弹红字）；
- 捕获其它异常 → 变成一条 `"Error: ...，你可以再试一次"` 的助手消息显示，**而不是让界面崩**。

```python
# agent_runner.py（简化）
except Exception as error:
    if isinstance(error, AbortError):
        pass                                  # 主动取消，静默
    else:
        显示一条 "Error: ... 你可以再试一次"   # 其它错，友好提示
```

## 三、Token 记账

最后一个收尾话题：trendpower 一路在数 token（成本/触发压缩都靠它）。

- 每次模型回复都带 `usage`（`promptTokens`/`completionTokens`），由各 provider 的 `_to_token_usage` 算出来。
- 第 7 课强调过的坑：开缓存时要把 `cache_read` 加回 `promptTokens`，否则压缩失灵——这个折算同时做在"一次性返回"（`model_provider.py:_to_token_usage`）和"流式累计"（`stream_utils.py:StreamAccumulator`）两处，保证两条路径数出来一致。

## 四、④ 小结

- **取消**靠一个 `AbortSignal` 端到端串起：用户 → 循环 → 工具 → 子进程 → 子 agent，全是"监听同一信号、各自优雅退出"。
- **失败三件套**：工具错→喂回模型自愈；撞上限→软着陆交卷；压缩失败→继续不压缩。共同哲学是"辅助不连累主线、可恢复就喂回、到头了优雅收"。
- 前端再兜一层：取消静默、其它错变友好提示。
- Token 记账两条路径一致，且和缓存正确折算。
