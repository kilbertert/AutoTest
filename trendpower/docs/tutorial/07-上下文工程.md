# 第 7 课 · 上下文工程 ⭐

这是全教程**含金量最高**的一课。前面 02–06 的铺垫，都是为了让你能读懂这里。

主题只有一句话：**模型每走一步，都要把"一整桌纸"重读一遍——而这桌纸又贵、又有上限、还会乱。上下文工程就是把这桌纸经营好。**

---

## 一、根本约束：模型没记忆，每步重发整段对话

### 先理解这件反直觉的事

模型是**无状态的**——它不记得你们上一句聊了啥。所以 agent 每走一步 `_think`，都必须把 **系统提示 + 从头到现在的整段对话 + 工具清单** 全部重新发过去，它才知道"在干嘛、到哪了"。

回到第 4 课那个循环，假设一个任务走了 7 步：

- 第 1 步发：系统提示 + 工具清单。
- 第 2 步发：系统提示 + 工具清单 + 第 1 步全部来回。
- …
- 第 7 步发：把前 6 步**全部重发一遍**。

那段开头不变的"系统提示 + 工具清单"，被**一字不差重发了 7 次**；越往后每步要重读的越多。

### 代码：重发就发生在这里

```python
# foundation/models.py:Model._build_provider_params
messages = []
if prompt:                                   # 系统提示
    messages.append({"role": "system", "content": [{"type": "text", "text": prompt}]})
messages.extend(context.get("messages"))     # + 整段对话（每步都是全量）
```

每次 `_think` 都调一遍这个，`context["messages"]` 是当前的**完整**对话。这就是"重发整桌纸"。

### 这桌纸的三重压力

- **有上限**：模型一次能看的 token 有限（桌子只摆得下这么多张纸），堆满就"看不全"。
- **很贵**：按 token 收费，重复发的前缀重复计费。
- **会乱**：纸越多噪音越多，模型注意力越容易被带偏。

> **token** = 模型计费和计量的单位，约等于"一小段文字"（英文 ~4 字符/token）。记住：token 越多，越贵越慢。

下面是 trendpower 经营这桌纸的几招。

---

## 二、第 1 招：稳定前缀 + 缓存——把"重发不变前缀"的成本压掉

### 设计

既然开头那段每步都重发，那就：① 把**所有不变的内容**死死堆在最前面，形成一个"每次一模一样的开头"；② 用 **prompt caching（提示缓存）**——告诉模型厂商"这段开头我上次发过、这次一字没变，直接复用上次的计算结果，给我打折"。

打个比方：那个失忆助手每天要听的"项目背景"一字不变，聪明做法是**印成一张卡片塞他兜里**，他自己看，你不用每天口述。缓存就是这张卡片。

### 代码：在三个稳定边界打"缓存标记"

请求的字节顺序是 `工具 → 系统提示 → 对话`。越靠前越稳定。trendpower 在三个边界打缓存断点：

```python
# community/anthropic/model_provider.py:_apply_cache_control（简化）
@staticmethod
def _apply_cache_control(system, anthropic_tools, anthropic_messages):
    if anthropic_tools:
        anthropic_tools[-1]["cache_control"] = _EPHEMERAL    # ① 缓存整个工具清单
    if anthropic_messages:
        last = anthropic_messages[-1]["content"]
        last[-1]["cache_control"] = _EPHEMERAL               # ③ 缓存到"对话最后一块"
# ② 系统提示的缓存标记在调用处单独打
```

- **①** 标记最后一个工具 → 缓存住整份工具清单（整轮对话都不变）。
- **②** 标记系统提示 → 缓存住系统提示（含下面要讲的 AGENTS.md）。
- **③** 标记"对话的最后一块" → 让**不断增长的对话**也走增量缓存：这一轮把到目前为止的对话写进缓存，下一轮就能命中。

效果：开头那一大段（工具+系统提示+项目说明）是请求里最大的不变部分，缓存住后**输入成本大降、响应更快**。这是性价比最高的一招。

### ⚠️ 一个隐藏的坑：缓存会让"压缩"失灵

这是最容易踩、也最能体现"牵一发动全身"的地方：

开了缓存后，模型返回的 `input_tokens`（输入 token 数）**只统计没命中缓存的部分**，命中的那部分被单独列在 `cache_read_input_tokens` 里。问题来了——**下面第 3 招的"压缩"，靠 `promptTokens` 判断对话是不是太长了**。如果 `promptTokens` 因为缓存突然"变小"，压缩就**永远不触发 → 最终撑爆窗口**。

所以 trendpower 在统计时，把缓存的 token **加回去**，保持 `promptTokens` = "这次发的真实大小"：

```python
# community/anthropic/model_provider.py:_to_token_usage（简化）
prompt_tokens = input_tokens + cache_read + cache_creation   # ★缓存部分加回来★
```

> 教学点：缓存（第1招）和压缩（第3招）看似不相干，却通过 `promptTokens` 这个数字耦合在一起。这种"远处的两块通过一个共享量纠缠"的情况，是真实工程里最隐蔽的 bug 来源。

> 对豆包/OpenAI 这类**自动缓存**的模型，它们的 `prompt_tokens` 本就含缓存部分，所以没这个坑；而第1招的"稳定前缀"对它们同样有效——开头不变，它们就自动命中缓存。

## 三、第 2 招：AGENTS.md 放进系统提示，而不是当第一句对话

### 背景

很多项目根目录有个 `AGENTS.md`，写着"这项目是干嘛的、有什么规矩"。trendpower 启动时自动加载它。**放哪喂，有讲究。**

### 设计：放进"系统提示"区

```python
# coding/agents/lead_agent.py（简化）
agents_section = ""
if agents_file.exists():
    agents_section = "\n<project_instructions>\n" + agents_content + "\n</project_instructions>\n"

prompt = (
    "<agent ...>...</agent>\n"
    "<tool_usage>...</tool_usage>\n"
    "<subagents>...</subagents>\n"
    f"{agents_section}"          # ★拼进系统提示，而不是塞进对话★
)
```

### 为什么这样更好（两个实在的理由）

1. **能被缓存**：它进了"不变的系统提示"（第1招的稳定前缀），于是被缓存住，省钱。
2. **不被压缩误伤**：第3招的压缩只改写**对话**，从不动系统提示。放在系统提示里，它就永远安全地待着，不会在某次压缩里被摘掉。

打个比方：项目背景卡片，该塞进助手兜里**长期带着**（系统提示），而不是写在某张随时会被清理的便利贴上（对话）。

## 四、第 3 招：压缩——桌子快满了，把中间旧草稿缩成一页便签

### 设计

对话快到阈值（默认约 10 万 token）时触发压缩：
- **保住开头**：那是任务目标，丢了就忘了在干嘛。
- **保住最近几轮**：那是"现在到哪了"，必须原样留。
- **把中间一大段**交给模型**总结成一小段**，原地替换。

打个比方：桌上草稿堆成山，你不会全扔，而是把**中间那堆旧草稿归纳成一页要点便签**压在桌角；最关键的"任务书"（开头）和"手头正写的几页"（最近）原封不动。

### 代码：切成"头/中/尾"

```python
# agent/compaction/compaction.py:create_compaction_middleware（简化）
async def before_model(params):
    messages = params["agentContext"]["messages"]
    if estimate_tokens(messages) < trigger_tokens:    # 还没到阈值，不动
        return None
    plan = plan_compaction(messages, keep_head, keep_recent)   # 切成 头/中/尾
    head, middle, tail = plan
    summary_text = await summarizer(middle, signal)   # 让模型把"中"总结成一段
    messages[:] = [*head, 一条装着summary的消息, *tail]  # 原地替换掉中段
```

- **`estimate_tokens`**（同文件）：估算对话多大。优先用模型上次返回的真实 `promptTokens`，没有就用"字符数÷4"的粗算，**取两者较大值**——这样"刚发生一波大工具结果"也能立刻触发，不必等下一次模型响应。（这就是第1招那个坑的另一端：它命脉系于 `promptTokens`。）
- **总结提示**要求保留：目标、改过的文件及状态、命令及结果、决策及理由、未解问题、下一步——确保"压缩"不丢关键信息。

### 第 3 招的铁律：问答必须成对，不能撕半张

这是压缩里最容易翻车的地方，单独讲。

**背景**：模型调工具，对话里会留下成对的两条：一条"我要调 read_file"（**问**，`tool_use`），紧跟一条"这是文件内容"（**答**，`tool_result`）。模型厂商**硬性要求这一对必须同时在、不能只留一半**，否则请求被拒。

**坑**：压缩要"砍掉中间一段"。要是切口正好把某对问答劈成两半——保住了"问"、把"答"压进了总结——就留下一个落单的半截，请求废了。

**怎么避开**：

```python
# agent/compaction/compaction.py:plan_compaction（简化）
# 不让"头"结束在一个 tool_use 上（否则它的答会被压走 → 落单）
while head_end > 0 and _has_tool_use(messages[head_end - 1]):
    head_end -= 1
# 不让"尾"从一条孤儿 tool 消息开始（它的问被压走了 → 落单）
while tail_start < n and messages[tail_start]["role"] == "tool":
    tail_start += 1
```

打个比方：**撕日历得一整张一整张地撕**，不能从中间斜着撕掉半张——那张就废了。一对问答就是"一整张"。

### 还有个兜底：压缩失败也不许把任务搞崩

```python
try:
    summary_text = await summarizer(middle, signal)
except Exception as exc:
    _log.warning("压缩失败，继续不压缩: %s", exc)
    return None                  # ★绝不让"压缩"反而搞崩正常任务★
```

## 六、招式全景

| 招 | 大白话 | 解决三重压力的哪个 |
|---|---|---|
| 稳定前缀 + 缓存 | 不变背景印成卡片，别每次重抄 | 贵（省钱）、慢（提速） |
| AGENTS.md 进系统提示 | 卡片塞兜里长期带 | 贵 + 不被压缩误伤 |
| 压缩 | 桌满了把旧草稿缩成便签 | 有上限（防撑爆） |
| 问答成对 | 撕日历整张撕 | 正确性（请求不废） |
| 工具结果裁剪 | 汇报只说重点 | 会乱（控增长） |

**一条主线**：把上下文当成一份**有限、昂贵、承载注意力的预算**来经营——该省的省（缓存）、该缩的缩（裁剪、压缩）、放对位置的放对（AGENTS.md），还要小心招式之间的隐藏耦合（缓存↔压缩）。

---

## 七、④ 小结

- 一切根源：**模型无记忆，每步重发整桌纸**（`_build_provider_params`）。
- 缓存把不变前缀的成本压掉，但要记得把缓存 token **加回 `promptTokens`**，否则压缩失灵。
- AGENTS.md 进系统提示 → 能缓存 + 不被压缩动。
- 压缩"保头保尾摘中段"，且**绝不劈开问答对**，失败还要兜底不崩。

还有一招比"事后压缩"更彻底——**根本不让噪音进主桌子**：派子 agent 去别的桌子上干。这属于"能力扩展"，下一课讲。
