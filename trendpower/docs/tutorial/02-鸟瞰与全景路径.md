## 一、鸟瞰：trendpower 是什么形状

### 1）一句话定位

trendpower 是一个**搭"ReAct 风格 agent 循环"的 Python 库**。ReAct = Reason + Act，"边推理边行动"。说人话：让模型**想一步、做一步、看结果、再想**，直到把活干完。

### 2）核心心智模型：大脑 vs 手脚

请把这个比喻刻在脑子里，整套教程都靠它：

> **模型（Claude/GPT/豆包）是大脑**——很聪明，但只会"输入文字、输出文字"，碰不到你的电脑。
> **trendpower 是手脚和规矩**——给大脑配上工具（读写文件、跑命令）、流程（循环）、和规矩（审批、上限）。

这套"手脚和规矩"有个行业叫法：**harness**（驾驭层）。

### 3）三个包，单向依赖

trendpower 的代码分成三个可独立安装的包，依赖**只能从左往右**：

```
trendpower-web  ──依赖──▶  trendpower-tui  ──依赖──▶  trendpower (trendpower-py)
（浏览器可视化）       （终端界面）          （核心库，不依赖任何上层）
```

| 包 | 是什么 | 角色 |
|---|---|---|
| **trendpower**（`trendpower-py/`） | 核心库 | agent 循环、工具、上下文管理全在这。**不知道界面的存在。** |
| **trendpower-tui** | 终端界面 | 把核心库接到一个终端 UI 上（你平时敲 `trendpower` 跑的就是它）。 |
| **trendpower-web** | 浏览器可视化 | 在 TUI 基础上，把"每轮发给模型的内容"实时画到浏览器里。 |

**为什么这么切**（设计合理性）：核心逻辑**完全不依赖界面**，所以同一套大脑+手脚，既能包成终端工具，也能包成网页可视化，将来还能包成别的。**界面是可替换的外壳，内核是稳定的。** 这条"单向依赖"是整个项目最重要的结构纪律，第 11 课会再回来谈。

### 4）核心库内部的四层

`trendpower` 内部又分四层，同样是**下层不依赖上层**：

```
coding     ← 编程专用的 agent 和工具（bash、读写文件…）
  ▲
agent      ← 通用的 ReAct 循环、中间件（不懂编程，只懂"想-做"）
  ▲
foundation ← 最底层的基本概念：消息、模型、工具的定义
  ▲
community  ← 第三方集成（OpenAI / Anthropic / MCP 适配器）
```

- `foundation`：基本词汇（第 3 课）。
- `agent`：发动机和配件槽（第 4、5 课）。
- `coding`：把发动机装上"编程工具"，组装成真正能用的 agent（贯穿后续）。
- `community`：对接各家模型和外部工具（第 7 课）。

## 二、全景路径：跟一次提问走一遍

现在顺着"你敲 `trendpower`、提个问题、看到回答"这条主干道走一遍。**每一站先记住"它在干嘛"，细节后面有专门的课。**

### 站 1：启动 —— 把 agent 组装出来

你敲 `trendpower`（入口 `trendpower-tui/trendpower_tui/__main__.py:main`），终端界面启动（`app.py`）。启动时最关键的一步，是把一个**装配齐全的 agent 造出来**：

```
app.py:_try_create_runner
  → 选模型 provider（Anthropic / OpenAI-兼容，含豆包）
  → create_coding_agent(...)        ← coding/agents/lead_agent.py，组装的核心
        把"系统提示 + 12 个工具 + 4 个中间件 + 模型"拼成一个 Agent
```

`create_coding_agent`（`coding/agents/lead_agent.py`）就是"大脑 + 手脚 + 规矩"的总装车间——后面好几课都会回到它。

### 站 2：你输入问题

你在输入框敲下问题回车，界面把它交给一个**桥接器** `AgentRunner`（`trendpower-tui/.../tui/agent_runner.py:submit`）。这个桥接器的职责，就是把"核心库吐出来的事件"翻译成"界面能渲染的消息"。

### 站 3：进入心脏 —— ReAct 循环

桥接器调用 `agent.stream(你的消息)`（`agent/agent.py:Agent.stream`），正式进入发动机。还记得第 1 课吗？`stream` 是个**异步生成器**，它一边推进"想→做"循环，一边 `yield` 事件出来：

```
循环 {
   想（think）：把"系统提示+整段对话+工具清单"发给模型，模型回话（可能要求调工具）
   若没要求调工具 → 结束，这就是最终回答
   做（act）：并行执行模型要的工具，把结果追加进对话，喂回去
}  直到模型不再调工具，或撞上限
```

### 站 4：结果流回界面

`stream` 每 `yield` 一个事件，桥接器（站 2 那个）就接住：
- `progress` 事件 → 更新"思考中… / 正在跑 xxx…"的指示器；
- `message` 事件 → 把助手回复、工具结果渲染到对话区。

为了不让界面被刷爆，桥接器**每 50 毫秒批量刷新一次**（`agent_runner.py` 的 `FLUSH_INTERVAL_SECONDS`），而不是每个事件都重画。

### 一图串起来

```
你敲 trendpower
  → __main__.main → 启动 TUI → _try_create_runner → create_coding_agent  ⟹ 造出 Agent
  → 你提问 → AgentRunner.submit → Agent.stream
       → 循环{ 想: 发给模型 ; 做: 跑工具 } 直到模型不再调工具
  → 事件流 → AgentRunner（50ms 批量）→ 界面渲染
```

> 想看这条路径"具体经过哪些文件、哪一行"，有一篇专门的追踪：[execution-flow.md](../execution-flow.md)。本课只要你记住这四站的**职责划分**即可。

## 三、本课小结

- trendpower = **大脑（模型）+ 手脚和规矩（harness）**。
- 三个包**单向依赖**（web→tui→core），核心库不知道界面存在——这是最重要的结构纪律。
- 核心库内部四层：`community / foundation / agent / coding`，下层不依赖上层。
- 一次提问的主干道四站：**组装 agent → 接收输入 → ReAct 循环 → 事件流回界面**。
