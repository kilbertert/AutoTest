# 第 5 课 · 中间件（配件槽）

上一课的发动机很纯粹——只懂"想-做"。那"压缩历史、注入技能、提醒待办、危险操作审批"这些行为是怎么加上去的？答案是**中间件**。这一课讲清这套可插拔机制，并逐个看挂在上面的四个中间件。

---

## 一、① 要解决什么问题

有些行为**不属于某个具体工具**，而是"每一步都要顺带做一下"的横切任务，比如：
- 对话太长了，发给模型前先压缩；
- 把可用的"技能清单"注入进系统提示；
- 危险操作执行前先弹窗问用户。

如果把这些全焊死在循环里，循环会越来越臃肿、改一个动全身。

---

## 二、② 设计决策：洋葱式"检查站"

trendpower 的决定：把这些行为做成一个个**可插拔的中间件**，挂在循环的固定时点上。

打个比方：**机场安检流程**。值机、安检、登机口各是一个独立检查站，"旅客"（也就是每一步的请求）依次经过。想加一道"测体温"，新设一个站就行，不用推倒整个流程。

三个设计要点：

1. **固定的挂载时点**：循环在几个关键时刻回调中间件——
   - `beforeAgentRun` / `afterAgentRun`：整轮对话开始/结束；
   - `beforeAgentStep` / `afterAgentStep`：每一步开始/结束；
   - `beforeModel` / `afterModel`：**发给模型前/后**（最常用）；
   - `beforeToolUse` / `afterToolUse`：执行工具前/后。
2. **顺序即语义**：中间件是个**有序列表**，谁先谁后是有讲究的（下面"压缩排最前"会看到）。
3. **靠共享上下文传话**：钩子返回的字典会被合并进一个共享的 `AgentContext`，中间件之间通过它通信，不用互相认识。

---

## 三、③ 看代码

### 中间件长什么样：一个"有钩子的对象"

还记得第 1 课 A5 的鸭子类型吗？中间件不需要继承任何基类，**只要是个"带某些钩子方法的对象"**即可。所以每个中间件工厂就地拼一个 `SimpleNamespace` 返回：

```python
# 比如 compaction 中间件结尾
return SimpleNamespace(beforeModel=before_model)   # 一个只有 beforeModel 钩子的对象
```

### 循环怎么调它们："按名字找钩子"

```python
# agent/agent.py:_before_model（分发逻辑，简化）
async def _before_model(self, model_context):
    for mw in self.middlewares:
        hook = getattr(mw, "beforeModel", None)    # 这个中间件有 beforeModel 吗？
        if hook is None:
            continue                               # 没有就跳过
        result = await hook({"modelContext": model_context, "agentContext": self._context})
        if result:
            model_context.update(result)           # 钩子返回的东西合并进上下文
```

`getattr(mw, "beforeModel", None)`——有这个钩子就调、没有就跳过。**这就是鸭子类型**：你"有哪个钩子"，就自动参与哪个时点。八个时点（`_before_model`、`_before_tool_use`…）的分发逻辑长得都一样。

### 顺序在哪定：总装车间

```python
# coding/agents/lead_agent.py（简化）
middlewares = [create_skills_middleware(skills_dirs), todo_middleware]
if enable_compaction:
    middlewares.insert(0, create_compaction_middleware(...))   # ★插到最前★
if ask_user is not None:
    middlewares.append(create_coding_approval_middleware(...))
```

**注意 `insert(0, ...)`**：压缩中间件被**刻意放到最前面**。因为后面的中间件（比如技能注入）要读对话内容，得先让压缩把过长的历史压实了，它们读到的才是"将要真正发出去"的样子。**顺序携带了设计意图。**

## 四、挂在上面的四个中间件（逐个看）

### 1）压缩（compaction）—— 对话太长就摘要

- **干嘛的**：对话快撑爆模型窗口时，把中间一大段总结成一小段。
- **挂哪个钩子**：`beforeModel`（发给模型前才需要）。
- **为什么排最前**：见上。
- 这是第 **07 课**的主角，那里细讲。文件：`agent/compaction/compaction.py`。

### 2）技能注入（skills）—— 把"能力目录"塞进系统提示

- **干嘛的**：在系统提示里注入一份"你有哪些技能可用"的目录（只给目录，不给正文）。
- **挂哪个钩子**：`beforeAgentRun`（开场时扫描有哪些技能）+ `beforeModel`（每次发给模型前把目录拼进提示）。
- 细节在第 **08 课**。文件：`agent/skills/skills_middleware.py`。

### 3）待办系统（todo）—— 还会"反过来提醒模型"

这个最有意思，它**既是工具、又是中间件**：

```python
# agent/todos/todos.py:create_todo_system（返回一个工具 + 一个中间件）
def create_todo_system():
    store = []                    # 这次会话的待办列表（闭包记住，第1课A6）
    ...
    async def before_model(params):
        state.steps_since_last_write += 1
        if (len(store) > 0 and 已经很多步没更新待办了):
            return {"prompt": params["modelContext"]["prompt"] + _format_reminder(store)}
            # ↑ 在系统提示后面追加一段"你好久没更新待办了，当前待办如下…"
        return None

    async def after_tool_use(params):
        if params["toolUse"]["name"] == "todo_write":
            state.steps_since_last_write = 0   # 模型刚更新过，计数器清零
        return None

    return tool, SimpleNamespace(beforeModel=before_model, afterToolUse=after_tool_use)
```

- **工具那半**（`todo_write`）：模型可以调它来"建/改待办列表"。
- **中间件那半**：每步计数，**如果模型很多步没碰待办了，就在提示里塞一句提醒**；模型一旦更新了待办（`afterToolUse` 看到 `todo_write`），计数清零。
- 设计巧思：用 `beforeModel` 注入提醒 + `afterToolUse` 重置计数，让模型在长任务里**不会忘了维护进度清单**。

> （如果你用过类似工具看到过"todo 工具最近没用，要不要更新"的提示，就是这套机制干的。）

### 4）审批（approval）—— 危险操作先签字

- **干嘛的**：执行 `bash`/写文件/改文件这类危险工具前，弹窗问用户。
- **挂哪个钩子**：`beforeToolUse`（执行前拦截）。
- **看代码**：

```python
# coding/permissions/coding_approval_middleware.py:before_tool_use（简化）
async def before_tool_use(params):
    tool_use = params["toolUse"]
    if tool_use["name"] not in requires_approval:   # 不在危险名单 → 放行
        return None
    if tool_use["name"] in 已记住允许的:             # 之前选过"总是允许" → 放行
        return None
    decision = await ask_user(tool_use)              # 弹窗问
    if decision == "deny":
        return {"__skip": True, "result": "User denied..."}   # ★短路：不执行，把"被拒绝"喂回模型★
    if decision == "allow_always_project":
        await persist_allowed_tool(...)              # 记住，下次不再问
    return None
```

两个设计点：
- **危险名单**写死在 `requires_approval.py:CODING_TOOLS_REQUIRING_APPROVAL` = `bash / write_file / str_replace / apply_patch / mkdir / move_path`。加一个工具进审批，只是改这个列表。
- **`__skip` 短路协议**：用户拒绝时返回 `{"__skip": True, ...}`，循环识别这个标记后**跳过真正执行**，直接把"用户拒绝了，请换方案"作为结果喂回模型——**拒绝也是一条优雅的观测，而不是异常**。
- "总是允许"通过 `ApprovalPersistence` 存进项目设置，把人类信任**沉淀**下来。

---

## 五、④ 小结

- 中间件 = 挂在循环固定时点上的**可插拔检查站**；循环靠"按名字找钩子"调用它们（鸭子类型）。
- **顺序即语义**：压缩 `insert(0)` 排最前，保证后续中间件读到的是压缩后的对话。
- 四个中间件：**压缩**（第7课）、**技能注入**（第8课）、**待办**（工具+提醒中间件二合一）、**审批**（`beforeToolUse` 拦截 + `__skip` 短路）。
- 想给 agent 加新行为？写一个带相应钩子的 `SimpleNamespace`，插进 `lead_agent.py` 的列表即可（第 11 课有速查表）。
