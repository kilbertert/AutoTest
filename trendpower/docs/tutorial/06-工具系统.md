# 第 6 课 · 工具系统

第 3 课认识了 `Tool` 这个名词，第 4 课看到循环怎么"调"工具。这一课解剖工具系统的**内部**：一次工具调用，从参数到结果，到底走了哪几道工序。

---

## 一、① 要解决什么问题

模型调工具时会**自己生成参数**（读哪个文件、写什么内容）。两个麻烦：
1. 这些参数**不可信**——可能漏字段、类型错、瞎编一个不存在的参数。直接拿去执行，轻则报错、重则误操作。
2. 工具结果**形态各异**——有的返回一长串、有的返回报错。怎么统一地"汇报给模型"，让它能据此决策？

工具系统就是来解决这两件事的：**入口处把不可信参数驯服，出口处把结果规整成统一协议。**

---

## 二、② 设计决策

1. **入口：pydantic 当"校验边界"。** 参数先用 pydantic 校验，过了才交给业务代码；没过就把"哪里错了"喂回模型让它重填。
2. **出口：结构化结果当"通信协议"。** 工具返回 `{ok, summary, data}` 或 `{ok:false, error, code}`，并把错误码归类成"错误类型"。
3. **回传前再裁剪。** 不同工具的结果按各自策略截断/精简，控制它占多少上下文（这一条直接服务于第 7 课的上下文工程）。

---

## 三、③ 看代码

### 工序 1：定义工具 + 校验边界

```python
# foundation/tools.py:define_tool（简化）
def define_tool(*, name, description, parameters, invoke):
    async def wrapped(raw_input: dict, signal=None):
        validated = parameters.model_validate(raw_input)   # ★校验边界★：原始dict → 校验过的对象
        return await invoke(validated, signal)             # 业务代码拿到的永远是合法对象
    return FunctionTool(name=name, description=description,
                        parameters=parameters, invoke=wrapped)
```

关键就是 `parameters.model_validate(raw_input)` 这一行：
- 模型吐出来的是个原始字典（不可信）；
- pydantic 拿参数 schema 一校验——缺字段/类型错就在这里抛错；
- 过了，业务函数 `invoke` 拿到的就是**保证合法**的对象。

校验失败抛的错，会被第 4 课 `_act` 里的 try 接住、变成 `"Error: ..."` 喂回模型，让它自己改正重填——**而不是崩**。这就是"表填错了当场打回"。

举个真实工具的参数定义：

```python
# coding/tools/read_file.py
class _ReadFileParams(BaseModel):
    description: str = Field(description="解释你为什么要读这个文件。")
    path: str = Field(description="要读的文件的绝对路径。")
    startLine: Optional[int] = Field(default=None, ge=1)   # 可选，且必须 ≥1
    endLine: Optional[int] = Field(default=None, ge=1)
```

`Field(description=...)` 里的描述会变成给模型看的"参数说明书"。`ge=1`（大于等于1）这种约束，pydantic 会在校验时自动检查。

### 工序 2：把结果规整成统一协议

工具执行完，结果在追加进对话前，过一道规整（`agent/tool_result/runtime.py`）：

```python
# agent/tool_result/runtime.py:normalize_tool_result（简化思路）
def normalize_tool_result(result):
    if 是结构化成功:  return {"ok": True, "summary": ..., "data": ...}
    if 是结构化错误:  return {"ok": False, "summary": ..., "error": ...,
                            "errorKind": infer_tool_error_kind(code)}   # ★把错误码归类★
    if 是 "Error:" 开头的字符串:  当作未知错误处理
    其它:  当作成功，结果转成字符串塞进 summary
```

`infer_tool_error_kind` 把错误码翻译成"错误类型"，让模型一眼看懂：

```python
# agent/tool_result/runtime.py
def infer_tool_error_kind(code):
    if code.startswith("INVALID_"):     return "invalid_input"      # 参数非法
    if code.endswith("_NOT_FOUND"):     return "not_found"          # 没找到
    if code == "RG_NOT_FOUND":          return "environment_missing"# 环境缺工具
    if code.endswith("_FAILED"):        return "execution_failed"   # 执行失败
    return "unknown"
```

**为什么有用**：系统提示里要求模型"用错误码决定下一步"。比如看到 `not_found` 就该换个路径，看到 `invalid_input` 就该改参数——而不是对着一坨乱码瞎猜。

### 工序 3：按工具裁剪（控制占多少上下文）

```python
# agent/tool_result/policy.py:get_tool_result_policy（节选）
def get_tool_result_policy(tool_name):
    if tool_name in ("list_files", "glob_search", "grep_search", "file_info", ...):
        return {"preferSummaryOnly": True, "includeData": False, "maxStringLength": 1000}
        # ↑ 列目录/搜索：只留摘要，丢掉原始数据，最多1000字符
    if tool_name == "read_file":
        return {"includeData": True, "maxStringLength": 12000}      # 读文件：留得多
    if tool_name == "task":
        return {"preferSummaryOnly": True, "maxStringLength": 8000}  # 子agent报告：留较多
    return {"includeData": True, "maxStringLength": 4000}           # 默认
```

**设计思想**：喂回模型的每个字符都是成本和注意力。所以**按工具的信息密度分别对待**——
- 列目录/搜索：模型要的是"找到了什么"这个结论，原始长列表丢掉（`preferSummaryOnly`）。
- 读文件：你读就是为了看内容，留得多（12000 字符）。
- 子 agent 报告：是辛苦查一圈的浓缩结论，值得多留（8000）。

这一条是工具系统**伸向上下文工程的手**——下一课会从全局再讲一遍"为什么省、怎么省"。

## 四、12 个内置工具巡览

`coding/tools/` 下的 12 个工具，按用途分四组：

| 组 | 工具 | 干嘛 |
|---|---|---|
| **搜索/浏览**（只读） | `list_files`、`glob_search`、`grep_search`、`file_info` | 列目录、按通配符找文件、按内容搜、看文件信息 |
| **读** | `read_file` | 读文件（支持只读某几行） |
| **写/改**（要审批） | `write_file`、`str_replace`、`apply_patch`、`mkdir`、`move_path` | 整体写、替换片段、打补丁、建目录、移动 |
| **执行**（要审批） | `bash` | 跑 shell 命令 |
| **管理** | `todo_write` | 维护待办清单（第5课讲过） |

挑两个看看细节：

### `read_file`：读了为了看，所以截断点大

`coding/tools/read_file.py` 用 `pathlib` 读文件（第 1 课 A7），支持只读 `startLine~endLine` 某一段（大文件不必整读），默认上限 12000 字符。

### `bash`：跑命令，能被急停 kill

`coding/tools/bash.py` 用 `create_subprocess_exec` 启子进程（第 1 课 A2）。亮点是它和急停信号的配合：

```python
# coding/tools/bash.py（简化）
if signal is not None:
    def _kill():
        proc.kill()                     # 被急停时杀掉子进程
    remove_listener = signal.add_listener(_kill)   # 注册到急停信号上
try:
    stdout, stderr = await proc.communicate()      # 等命令跑完
finally:
    remove_listener()                   # 善后：解绑（第1课A7的 try/finally）
```

> 顺带一个**已知待改进点**（教学诚实）：现在 bash 没有超时、成功时只取 stdout、超长输出"留头丢尾"。这些在项目的优化清单里——说明再成熟的设计也总有继续打磨的余地。

## 五、④ 小结

一次工具调用的完整工序：
1. **入口校验**（`define_tool` + pydantic）：不可信参数被驯服，错了打回让模型重填。
2. **执行**：业务函数拿到合法对象干活，能被急停打断。
3. **出口规整**（`normalize_tool_result` + `errorKind`）：结果变成统一协议，错误能归类。
4. **裁剪**（`policy.py`）：按工具信息密度决定回传多少，控制上下文占用。
