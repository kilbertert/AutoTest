# trendpower 项目亮点与简历写法

> 本文两部分：
> 1. **系统亮点** —— 逐个拆解 trendpower 在 Harness 设计与上下文工程上的工程亮点，每个亮点按"解决什么问题 / 怎么设计 / 为什么是亮点"组织，可直接作为面试讲解脚本。
> 2. **简历写法** —— 如何把这个项目写进简历（含可直接套用的 bullet point、量化口径、项目命名建议）。

## 第一部分 · 系统亮点

### 项目一句话定位

> 一个从零实现的、生产级架构的 **ReAct 智能体框架（agent harness）**：核心库纯 Python、无界面依赖，配套终端 UI 与浏览器可视化；完整实现了上下文缓存、上下文压缩、子智能体隔离、协作式取消、多模型厂商适配等现代 agent harness 的核心机制。

客观规模（用于简历量化）：

| 指标 | 数值 |
|---|---|
| 核心库代码量 | ~6,100 行 Python（`trendpower-py`） |
| 全仓代码量 | ~10,600 行 Python（核心库 + 终端 UI + Web 可视化） |
| 内置工具数 | 16 个（bash / 文件读写 / 检索 / patch / 子 agent 等） |
| 单元/集成测试 | 9 个测试文件、46 个用例 |
| 支持模型厂商 | Anthropic、OpenAI，及一切兼容 OpenAI 接口的厂商（豆包/火山方舟、DeepSeek、Qwen、Kimi、GLM…） |
| 架构分层 | 3 个包单向依赖 + 核心库内 4 层 |

---

### 亮点 1 · 分层架构与严格单向依赖

**解决什么问题**：agent 系统很容易写成"循环、工具、界面、模型 SDK 全耦在一起"的大泥球，导致换模型要改循环、加界面要动核心、无法单独测试。

**怎么设计**：

- 三个包**严格单向依赖**：`trendpower-web → trendpower-tui → trendpower(core)`，下层永不依赖上层。
- 核心库内部再分四层：`coding → agent → foundation`，第三方集成走横向的 `community`。
- 核心库**完全不知道界面是否存在**，可独立运行（`examples/`）；`agent` 层**领域无关**（不懂"编程"，只懂"想—做"），能驱动任意领域的 agent。

**为什么是亮点**：边界清晰带来的回报是可验证的——`trendpower-web` 想加"把每轮请求可视化"，只需在最薄的 provider 层继承重写 `stream`/`invoke`、广播请求，**完全不碰循环、工具、中间件**。这是"用对的抽象换来的扩展自由"的直接证据。

---

### 亮点 2 · 以异步生成器为骨架的 ReAct 循环（流式 + 哨兵值）

**解决什么问题**：agent 既要"看一步做一步"地推进多步任务，又要把中间过程（在思考、在调工具、产出消息）**实时**反馈给界面，不能等整个任务跑完才出结果。

**怎么设计**：主循环 `agent/agent.py:Agent.stream` 本身是一个**异步生成器**，一边推进 `think → act` 循环，一边 `yield` 出 `progress`/`message` 事件供前端 `async for` 即时渲染。内部用一个**哨兵事件** `{"type": "_think_done", ...}` 巧妙解决了"异步生成器既要流式 `yield`、又要交还最终结果"的语言限制（等价于 TS 的 `yield*` + `return value`）。

**为什么是亮点**：用 Python 原生异步原语干净地表达了"流式 agent 循环"这一并不平凡的控制流，且 `_act` 用 `asyncio.create_task` + `asyncio.wait(FIRST_COMPLETED)` 让同一轮的多个工具调用**真正并行**执行——并发是架构白送的，不是额外加的。

---

### 亮点 3 · 可插拔中间件：把横切关注点从主循环剥离

**解决什么问题**：压缩、技能注入、待办提醒、危险操作审批……这些行为横切在循环各处。若直接写进主循环，循环会迅速腐化成无法维护的巨型函数。

**怎么设计**：主循环只管"想—做"，在六个固定节点（`beforeModel`/`afterModel`/`beforeAgentRun`/`beforeToolUse`/`afterToolUse`…）广播事件。中间件是带相应钩子的对象，用 `getattr(mw, hook, None)` **鸭子类型**探测——只实现自己关心的钩子。三条设计约束很讲究：

- **顺序即语义**：中间件按列表顺序执行，压缩被 `insert(0)` 保证"在任何读对话的中间件之前先压好"。
- **返回值即修改**：钩子返回 dict 即被合并进上下文，这是影响主流程的唯一途径，边界清晰。
- **可组合**：四个自带中间件（压缩 / 技能 / 待办 / 审批）互不感知、自由增删。

**为什么是亮点**：这是一套教科书式的"用中间件解耦横切关注点"的设计，新增任意横切行为都不需要碰主循环——可直接对标 Web 框架的中间件模型来讲。

---

### 亮点 4 · 工具系统的"校验—执行—规整—裁剪"四段式

**解决什么问题**：工具的输入来自大模型，**不可信**（字段缺失、类型错、值非法）；工具的输出体量**不可控**（一次 `grep` 可能刷屏）。两头都需要驯化。

**怎么设计**：

1. **校验闸门**：`define_tool` 用 pydantic 在执行前强制校验模型传来的原始 dict，坏输入根本进不了执行体。
2. **执行**：执行体在"输入已合法"前提下干活，逻辑得以简化。
3. **规整**：结果统一成结构化形态——成功 `{ok, summary, data}`、失败 `{ok, summary, error, code}`，模型据此稳定判断成败。
4. **裁剪**：按工具类型分级裁剪后才进对话（列目录只留摘要、读文件留多些、子 agent 报告只留 8000 字摘要）。

还留了 `raw_input_schema` 逃生舱，让 MCP 外部工具直接透传自己的 JSON Schema，不经 pydantic 往返。

**为什么是亮点**：把"不可信输入 + 任意副作用 + 不可控输出"系统性地驯化成"可信、稳定、可控"的一条消息——这是 agent harness 健壮性的核心，体现了对"模型输出不可信"这一第一性原则的尊重。

### 亮点 5 · 上下文缓存，以及一个"牵一发动全身"的隐藏耦合 ⭐

**这是整个项目最有讲头、最能体现工程深度的亮点。**

**解决什么问题**：模型无状态，每步都要重发"系统提示 + 完整对话 + 工具清单"。开头那一大段不变的前缀被一字不差重发几十次，既贵又慢。

**怎么设计**：

- 把所有不变内容堆在请求最前形成**稳定前缀**，并在三个稳定边界打 Anthropic 缓存断点（工具清单尾、系统提示、对话最新一块），让不变前缀走缓存、增长的对话走**增量缓存**（`community/anthropic/model_provider.py:_apply_cache_control`）。
- 把项目级指令 `AGENTS.md` 放进**系统提示**而非首条用户消息——使它既进入可缓存前缀，又永不被压缩误删。

**真正的亮点在于发现并修复了一个隐蔽耦合**：开启缓存后，厂商返回的 `input_tokens` 只统计**未命中**缓存的部分。而上下文压缩靠 `promptTokens` 判断"对话是否过长"。若放任不管，`promptTokens` 会因缓存"虚假变小" → **压缩永不触发 → 最终撑爆上下文窗口**。修复是在 token 统计处把缓存部分加回去，并在"一次性返回"和"流式累计"两条路径保持一致：

```python
prompt_tokens = input_tokens + cache_read + cache_creation   # 缓存 token 加回来
```

**为什么是亮点**：这是"两个看似无关的模块通过一个共享数值远程耦合"的真实案例——缓存（省钱）和压缩（防撑爆）通过 `promptTokens` 纠缠。能定位并讲清这个 bug，直接证明对系统全局数据流的掌控力。面试时这一条最值得展开。

---

### 亮点 6 · 上下文压缩机制（保头保尾摘中段 + 问答对不可分割不变量）

**解决什么问题**：长任务的对话会无限增长，迟早撞上上下文窗口上限并撑爆。

**怎么设计**（`agent/compaction/compaction.py`）：对话超过阈值（默认约 10 万 token）时触发：

- **保住开头**（任务目标）+ **保住最近 8 轮**（当前进展），把**中间一大段**交给模型总结成一小段、原地替换。
- **触发判断取真实值与粗算的较大值**：优先用模型上次返回的真实 `promptTokens`，否则用"字符数 ÷ 4"粗算，**取较大值**——这样刚产生一波大工具结果也能立即触发，不必等下一次模型响应。
- **铁律：绝不劈开"问答对"**。`tool_use`（问）与 `tool_result`（答）必须成对存在，否则厂商拒绝整个请求。`plan_compaction` 因此调整切口：不让"头"结束在 `tool_use` 上、不让"尾"起始于孤儿 `tool` 消息，杜绝压缩切口把某对问答劈成两半导致请求作废。
- **失败兜底**：总结步骤若抛错，记日志后 `return None` 继续不压缩，**绝不让辅助功能搞挂主任务**。

**为什么是亮点**：压缩看似简单，真正难的是那条"问答对不可分割"的正确性不变量——这是只有真正跑通过 agent 才会踩到、并系统性解决的坑。它和"取较大值触发"一起，体现了对边界条件的扎实把握。

---

### 亮点 7 · 子智能体隔离（事前防污染，与压缩互补）

**解决什么问题**：压缩是"噪音堆上桌后再补救"。像"在整个项目里查清某功能怎么实现"这种活会翻几十个文件、产生大量中间噪音——能不能根本不让它上主桌？

**怎么设计**（`coding/tools/task.py` + `agent/subagent/runner.py`）：提供 `task` 工具，把脏活派给一个**独立子 agent**（自己的对话 + 受限工具集），干完只把**一句结论**汇报回主线，中间过程主线完全看不到。两种模式：`explore`（只读工具、免审批）与 `general`（完整工具、转发审批）。两条写死的安全红线：子 agent **永不含 `task`**（杜绝无限递归套娃）、**不含 `ask_user_question`**（后台任务不能反向弹窗阻塞用户）。并发白送——主循环 `_act` 本就并行，一次发 3 个 `task` 就是 3 个子 agent 同时干。

**为什么是亮点**：这是"事前隔离"与"事后压缩"互补的完整上下文治理方案，且复用了同一套循环（子 agent 就是再跑一遍 `Agent.stream`），体现了核心抽象的可复用性。

---

### 亮点 8 · 端到端协作式取消与失败分级

**解决什么问题**：交互式 agent 必须能随时打断且**立即**生效；长任务跑几十步、调一堆工具、连网等模型，出错是常态——它该"一出错就崩"还是"稳稳兜住"？

**怎么设计**：

- **协作式取消**：一个自定义 `AbortSignal` 把"打断"从用户一路串到底层——循环每步开头 `throw_if_aborted` 主动检查、`_act` 里和取消任务 `asyncio.wait` 赛跑、`bash` 注册 `proc.kill` 回调杀子进程、子 agent 注册 `inner.abort` 联动停止。没有谁被强杀，所有人监听**同一个信号**各自优雅退出。
- **失败分级**：工具出错 → 变成 `"Error: ..."` 消息喂回模型**自愈**；撞 `maxSteps=100` 上限 → **软着陆**（注入"请总结进度"提示并把 `tools` 设为 `None`，强制交一份交接总结而非崩溃）；压缩失败 → 退化继续不压缩。前端再兜一层：`AbortError` 静默、其它异常变友好提示。

**为什么是亮点**：健壮性不是事后补的，而是渗透在每个零件里的一致态度。"maxSteps 软着陆"尤其能讲——多数实现直接抛错丢弃全部工作，trendpower 让模型优雅交卷。

---

### 亮点 9 · 模型厂商抽象：一个 Protocol 接入全部国产模型

**解决什么问题**：模型厂商众多、接口各异，怎么不让厂商细节污染核心？

**怎么设计**：核心只依赖一个结构化类型 `ModelProvider`（Python `Protocol`，只要"长得像"——实现 `invoke`/`stream`——就是合法 provider，无需继承）。因此：接 Anthropic、OpenAI 各写一个类即可；而豆包/火山方舟、DeepSeek、Qwen、Kimi、GLM 等**兼容 OpenAI 接口的厂商连新类都不用写**，直接复用 `OpenAIModelProvider`、在配置里加个预设。

**为什么是亮点**：这是"面向抽象而非实现"的活教材——一个恰当的 Protocol，换来对整个国产大模型生态的零成本兼容。

---

### 亮点 10 · MCP 集成按"协议沉核心、接线留前端"拆分

**解决什么问题**：要接入业界标准的 MCP（Model Context Protocol）外部工具，但又不想让"进程生命周期、配置路径"这些前端细节污染核心库。

**怎么设计**：把 MCP 一分为二——可复用的**协议客户端**（连接、列工具、调工具、把 MCP 工具适配成 trendpower `Tool`）沉到核心 `community/mcp/`；**进程相关的接线胶水**（去哪找配置、绑界面生命周期、`/mcp` 命令）留在前端 `trendpower-tui/.../mcp/`，由它 `import` 核心那套。

**为什么是亮点**：又一次印证单向依赖原则——通用的下沉、进程特有的上浮，核心库保持纯净。

### 二、简历项目段落（可直接套用）

**trendpower — Production-Grade LLM Agent Harness**  *(Python, asyncio, Anthropic/OpenAI)*

> 从零设计并实现的 ReAct 智能体框架，含核心库、终端 UI 与浏览器可视化三个单向依赖的包，~10K 行 Python。

- 设计**异步生成器驱动的流式 ReAct 循环**，支持同一轮内多工具 `asyncio` 并行执行与边推进边渲染的事件流；以可插拔**中间件**架构将上下文压缩、技能注入、操作审批等横切关注点零侵入地解耦出主循环。
- 实现完整**上下文工程**栈：Anthropic 提示缓存（稳定前缀 + 三处增量缓存断点）、超阈值**对话压缩**（保头保尾摘中段，并保证 `tool_use`/`tool_result` 问答对不可分割的正确性不变量），将长任务上下文成本与撑爆风险显著降低。
- **定位并修复缓存与压缩间的隐藏耦合 bug**：缓存导致 `promptTokens` 失真使压缩永不触发，通过在两条 token 统计路径折算缓存命中量修复，保障长会话稳定运行。
- 构建**子智能体隔离**机制（`task` 工具）：将高噪音的检索类子任务委派给受限工具集的独立 agent，仅回传结论以保护主上下文；内置防递归、防阻塞两条安全不变量。
- 基于 Python `Protocol` 抽象模型厂商接口，一套抽象即兼容 Anthropic、OpenAI 及全部 OpenAI-兼容国产模型（豆包/DeepSeek/Qwen/Kimi/GLM）；实现端到端**协作式取消**与分级失败处理（工具错误自愈、步数上限软着陆、辅助功能失败降级不崩）。
- 沉淀 46 个单元/集成测试用例覆盖循环、压缩、子 agent、缓存等核心路径。

### 四、面试讲解建议

面试被追问时，**首推讲亮点 5（缓存↔压缩耦合 bug）和亮点 6（压缩的问答对不变量）**——这两条最能体现"真正跑通过 agent、对系统全局数据流有掌控"，远比罗列功能有说服力。讲法套路：

1. **背景约束**：模型无状态，每步重发整段上下文 → 又贵又会撑爆。
2. **两个对策本身**：缓存省钱、压缩防撑爆。
3. **冲突点（高光）**：缓存让 `input_tokens` 只算未命中部分 → `promptTokens` 失真 → 压缩判断失灵 → 永不触发。
4. **修复与验证**：两条统计路径折算缓存命中、保持一致；并解释为何"取真实值与粗算较大值"还能让大工具结果立即触发压缩。

这套"约束 → 设计 → 冲突 → 修复"的叙事，比任何功能清单都更能证明工程能力。
