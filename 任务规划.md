AutoGenesis 接口测试扩展方案

Context

AutoGenesis 目前是一个纯 UI 自动化测试框架（Appium + pywinauto），不支持接口/API 测试。用户需要在现有项目基础上扩展接口测试能力。经分析，BDD 基础设施（Gherkin 解析、Step 匹配、Copilot 集成）和 MCP
Server 架构模式可以直接复用，但需要新建一个 api-mcp-server 来提供 HTTP 测试工具。

设计目标

1. 新建 api-mcp-server/ — 遵循现有 MCP Server 架构模式
2. 扩展 bdd_ai_toolkit — 支持 API 测试的 MCP Server 安装和 Copilot 提示词
3. 新增接口测试专用工具：HTTP 请求、响应断言、变量提取、OpenAPI 解析
4. 复用 gen_code 模式生成 pytest + requests 测试代码

2.2 api_session.py — HTTP 会话管理

替代 DriverSessionManager，管理 HTTP 客户端会话：

class ApiSessionManager:
    """
    管理 HTTP 会话，包括：
    - base_url: API 基础地址
    - headers: 全局请求头
    - auth: 认证信息（token, api_key, basic_auth）
    - cookies: 持久化 Cookie
    - timeout: 请求超时
    - variables: 提取的变量（用于链式调用）
    """

    def __init__(self, config: dict):
        self.base_url = config.get("base_url", "")
        self.headers = config.get("headers", {})
        self.auth = config.get("auth", {})
        self.timeout = config.get("timeout", 30)
        self.variables = {}  # 存储提取的变量
        self.session = None  # requests.Session 或 httpx.Client

    def get_client(self) -> httpx.Client:
        """获取或创建 HTTP 客户端"""

    def update_headers(self, headers: dict):
        """更新全局请求头"""

    def set_variable(self, name: str, value: any):
        """存储提取的变量"""

    def resolve_variables(self, text: str) -> str:
        """替换字符串中的 {{variable}} 占位符"""

关键设计决策：使用 httpx 而非 requests，因为 httpx 支持 async/await（MCP 工具是 async 的），且 API 兼容 requests。

2.3 tools/http_tool.py — HTTP 请求工具

注册以下 MCP 工具：

┌─────────────┬─────────────┬──────────────────────────────────────────────┐
│   工具名    │    功能     │                     参数                     │
├─────────────┼─────────────┼──────────────────────────────────────────────┤
│ http_get    │ GET 请求    │ url, headers?, params?, extract_path?        │
├─────────────┼─────────────┼──────────────────────────────────────────────┤
│ http_post   │ POST 请求   │ url, body?, headers?, params?, extract_path? │
├─────────────┼─────────────┼──────────────────────────────────────────────┤
│ http_put    │ PUT 请求    │ url, body?, headers?, params?, extract_path? │
├─────────────┼─────────────┼──────────────────────────────────────────────┤
│ http_delete │ DELETE 请求 │ url, headers?, params?                       │
├─────────────┼─────────────┼──────────────────────────────────────────────┤
│ http_patch  │ PATCH 请求  │ url, body?, headers?, params?, extract_path? │
└─────────────┴─────────────┴──────────────────────────────────────────────┘

每个工具的返回格式：
{
  "status": "success",
  "data": {
    "status_code": 200,
    "headers": {...},
    "body": {...},
    "elapsed_ms": 150
  }
}

extract_path 参数：支持 JSONPath 表达式，从响应中提取值存入 session.variables，供后续请求使用（如提取 token）。

变量替换：所有 url/body/headers 中的 {{variable_name}} 会被替换为之前提取的变量值。

2.7 tools/gen_code_tool.py — 代码生成

复用现有的 before_gen_code → preview_code_changes → confirm_code_changes 三步模式。

生成的代码从 UI 自动化切换为 HTTP 测试：

# 生成的代码示例（替代 pywinauto/Appium 的 driver 操作）
import pytest
import requests

@pytest.fixture
def api_client():
    session = requests.Session()
    session.headers.update({"Content-Type": "application/json"})
    session.base_url = "https://api.example.com"
    yield session

def test_create_user(api_client):
    response = api_client.post("/users", json={"name": "test"})
    assert response.status_code == 201
    data = response.json()
    assert data["name"] == "test"
    user_id = data["id"]

    response = api_client.get(f"/users/{user_id}")
    assert response.status_code == 200

四、新增 behave-demo/features/api_demo.feature

示例 BDD API 测试用例：

Feature: 用户管理 API

  Background:
    Given API 基础地址为 "https://jsonplaceholder.typicode.com"

  Scenario: 获取用户列表
    When 我向 "/users" 发送 GET 请求
    Then 响应状态码应为 200
    And 响应体应为一个数组
    And 数组长度应大于 0
    And 响应体中的第一个用户的 "name" 字段不应为空

  Scenario: 创建新用户
    When 我向 "/users" 发送 POST 请求
      """
      {"name": "张三", "email": "zhangsan@example.com"}
      """
    Then 响应状态码应为 201
    And 响应体中的 "name" 应为 "张三"
    And 从响应中提取 "id" 存入变量 "new_user_id"

  Scenario: 使用提取的变量获取用户
    When 我向 "/users/{{new_user_id}}" 发送 GET 请求
    Then 响应状态码应为 200
    And 响应体中的 "id" 应为 "{{new_user_id}}"

2. BDD 层完全通用（bdd_ai_toolkit）

Gherkin 解析、Step 匹配引擎是与测试类型无关的。你可以写：

Feature: 用户登录接口
Scenario: 使用有效凭据登录
Given 请求头包含 "Content-Type: application/json"
When 我向 "/api/login" 发送 POST 请求
  """
  {"username": "admin", "password": "123456"}
  """
Then 响应状态码应为 200
And 响应体应包含 "token" 字段
And "token" 应为非空字符串

然后用 StepDefinitionLoader + StepMatcher 匹配到 Python 的 requests 实现。这套 BDD 基础设施完全可以复用。

3. Copilot 集成链路

VS Code → Copilot → MCP 协议 → MCP Server → 执行操作

这条链路把 MCP Server 换成接口测试的实现，就能让 Copilot 帮你调接口。
